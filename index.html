La intención es en realidad hacer modelos de programación diferentes BASADOS en hacer pruebas unitarias.


Primero hay que hacer las pruebas unitarias antes de hacer nada de código. Esto se basa en hacer que las pruebas unitarias fallen, haciendo un checklist

Pruebas, antes que nada


Por ejemplo, si por protocolo te piden crear una clase en cierta carpeta, primero corres pruebas unitarias para ver si la carpeta existe para empezar.


De este modo se va guiando al programa para que siga cierto tipo de acciones y protocolos.


Y así nos vamos yendo con los demás protocolos que deseemos que nuestro sistema cuente, como definir cuántos métodos queremos, qué tipo de valor de retorno queremos que devuelva. etc.


Así pues, las pruebas unitarias son algo que se va automatizando a lo largo del desarrollo, algo que ya está built-in en las features, no después de haberlas creado.

Las Ventajas


Las ventajas que esto conlleva es que a la hora de implementar nuevas features una vez que el sistema esté en una fase completa no sea necesario investigar o analizar de más el código.


Podemos primero definir la estructura a partir de los user stories mientras hacemos los test cases Así pues el desarrollador tiene en qué trabajar y al mismo tiempo resolver los test cases


Los Test cases en promedio agregan 10% de tiempo al desarrollo, lo cual puede ser algo pesado para trabajos con Deadlines cortos, pero vale la pena al mediano y largo plazo


Supogamos que nos entregan un sistema para que nosotros como desarrolladores le implementemos una nueva feature. Lo primero que haremos será correr las pruebas unitarias, y una vez que veamos que todas son exitosas, tendremos la certeza de que nos entregaron el código en un estado sólido, y al implementar nuestra feature y entregar el código al cliente, el tiempo que nos tomará asegurarnos de que todo está como nos lo dieron Más los features que implementamos será mínimo.

¿Cómo sabes si necesitas o no pruebas unitarias en tu código?


Digamos que no sabes a qué se refiere el valor de una variable, el valor que retorna una función, o un tipo de método. Normalmente harías un printr o verías en los logs el resultado. En lugar de eso, es mejor hacer una prueba unitaria, y este es el principal indicador de que tu sistema necesita pruebass unitarias.

¿Responsabilidad?, ¿Error de Comunicación?


Una de las ventajas de las pruebas unitarias es el deslinde de responsabilidad. Imagina que tu cliente o líder de proyecto te dice que lo que entregaste no fue lo que pidió. Y tu como desarrollador implementaste todo de acuerdo a los user stories y les realizaste las pruebas unitarias, es entonces que el problema se transparenta.

Tipos de reporte


Code coverage corre varias pruebas unitarias y enseña una gráfica con las pruebas unitarias que fueron success y las que arrojaron error. Y así se puede detectar con presición el porcentaje de funciones con pruebas unitarias aplicadas. Lo ideal es tener un 100% , pero está bien si tenemos un 80% o 90%.
